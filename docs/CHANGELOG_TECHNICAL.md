## الإصلاحات الثمانية الحرجة في الكود

### الإصلاح #1 — ثغرة IDOR في `deleteItem` و`editItem`
**الملف:** `server/routers/purchase/purchase-orders.router.ts`
**المشكلة:** الإجراء يتحقق من حالة/صلاحية `purchaseOrderId` المُرسل، لكن لا يتحقق أن `id` (الصنف) ينتمي فعلًا
لنفس ذلك الطلب. يسمح بحذف/تعديل صنف من طلب "ب" مكتمل عبر تمرير رقم طلب "أ" آخر قابل للتعديل.
**قبل:**
```js
const po = await db.getPurchaseOrderById(input.purchaseOrderId);
const item = await db.getPOItemById(input.id);
// لا يوجد تحقق من العلاقة بينهما
await db.deletePOItem(input.id);
```
**بعد:**
```js
if (item.purchaseOrderId !== input.purchaseOrderId) {
  throw new TRPCError({ code: "BAD_REQUEST", message: "هذا الصنف لا ينتمي لطلب الشراء المحدد" });
}
```
**لماذا:** بدون هذا التحقق، أي مستخدم بصلاحية عادية يستطيع تفريغ أي طلب من أصنافه أو تعديل بيانات طلب آخر مكتمل.

### الإصلاح #2 — حماية "آخر صنف متبقٍ" في `deleteItem`
**الملف:** نفسه أعلاه
**المشكلة:** يمكن حذف آخر صنف في طلب طالما حالته ضمن الحالات القابلة للتعديل، فيتحوّل الطلب لـ"رأس بلا بنود".
**الحل:**
```js
const allItemsInPO = await db.getPOItems(input.purchaseOrderId);
if (allItemsInPO.length <= 1) {
  throw new TRPCError({ code: "BAD_REQUEST", message: "لا يمكن حذف آخر صنف متبقٍ..." });
}
```
**لماذا:** هذا بالضبط الشكل الذي وُجد عليه الطلبات التسعة الفارغة أصلًا.

### الإصلاح #3 — شرط ميت في حذف الطلب الكامل
**الملف:** نفسه أعلاه (إجراء `delete`)
**المشكلة:** شرط الحماية كان يقارن `po.status` بقيم غير موجودة إطلاقًا في enum النظام
(`funded`, `partially_purchased`, `completed`) — فالشرط لا يتحقق أبدًا، ويستطيع أي `owner`/`admin` حذف أي طلب
مهما كانت حالته.
**الحل:** استُبدلت بقائمة `nonDeletableStatuses` تطابق فعليًا enum الحالات الحقيقي في `drizzle/schema.ts`
(`pending_accounting`, `pending_management`, `approved`, `partial_purchase`, `purchased`, `received`, `closed`).
**لماذا:** أخطر من ثغرة IDOR نظريًا — لا يحتاج أي حيلة، الحماية غير مفعّلة إطلاقًا مهما فعل المستخدم.

### الإصلاح #4 — تعطيل مسار الاستلام القديم غير الآمن (v1)
**الملف:** `server/routers/inventory/receipts.router.ts`
**المشكلة:** الصفحة الأمامية (`WarehouseReceive.tsx`) كانت مُعطَّلة (مُوجَّهة لـ `NotFound`)، لكن الإجراء
`receiveFromPurchase` بقي مسجَّلًا وقابلًا للاستدعاء المباشر عبر API، بلا أي تحقق من ملكية الصنف وبلا معاملة
ذرية (خلافًا للنسخة v2).
**الحل:** استُبدل جسم الإجراء بخطأ صريح يوجّه لاستخدام v2. الإجراءات الأخرى في نفس الملف (`scanBarcode` وغيرها)
بقيت سليمة لأنها لا تزال مستخدَمة فعليًا في صفحة نشطة (`WarehouseReturn.tsx`) — **لم يُحذف الراوتر بالكامل عمدًا.**
**لماذا:** تعطيل الصفحة الأمامية لا يُعطِّل الإجراء في الخادم. هذا فسّر جزءًا من حادثة انفصال روابط الاستلام.

### الإصلاح #5 — غياب المعاملة الذرية عند إنشاء الطلب
**الملفات:** `purchase-orders.router.ts` (مساري `create` و`saveDraft`)، `server/_core/db/purchase.ts`
**المشكلة:** إنشاء رأس الطلب وبنوده كانا استدعاءين منفصلين تحت autocommit. فشل إدراج البنود بعد نجاح إنشاء
الرأس يترك "رأسًا بلا أصناف" للأبد.
**الحل:**
- `createPurchaseOrder` و`createPOItems` أصبحتا تقبلان `tx` اختياريًا.
- `createPOItems` ترفض الآن مصفوفة فارغة (رمي خطأ) بدل التجاهل الصامت.
- كلا المسارين يُغلِّفان الإنشاء داخل `db.withTransaction(...)`.
**لماذا:** هذا يمنع تكرار نمط "رأس بلا بنود" من نقطة الإنشاء الأولى، بمعزل عن أي سبب لاحق آخر.

### الإصلاح #6 — الاستنتاج الخاطئ من قائمة فارغة (`every()`)
**الملفات:** `server/routers/inventory/inventory.router.ts`، `server/routers/inventory/receipts.v2.router.ts`
**المشكلة:** `[].every(predicate)` تُعيد `true` دائمًا في JavaScript (صدق فارغ). عند فحص "هل كل الأصناف
اكتملت؟" على طلب بصفر أصناف نشطة، كانت النتيجة `true`، فيتحوّل الطلب لحالة `received` رغم عدم وجود أي صنف.
**الحل:** إضافة `activeItems.length > 0 &&` قبل كل استدعاء `every()` كان يفتقده.
**لماذا:** هذا لا يُنتج المشكلة، لكنه كان يُخفي حدوثها بتحويل أي طلب فارغ (لأي سبب) لحالة "مُستلم" مضلِّلة.

### الإصلاح #7 — إضافة مفاتيح خارجية حقيقية (Foreign Keys)
**الملفات:** `drizzle/schema.ts` + `drizzle/migrations/critical_fix_07_*.sql`
**المشكلة:** لا يوجد أي قيد مرجعي في قاعدة البيانات نفسها بين:
- `purchase_order_items.purchaseOrderId → purchase_orders.id`
- `warehouse_receipt_items.purchaseOrderItemId → purchase_order_items.id`
- `inventory_transactions.purchaseOrderItemId → purchase_order_items.id`

**الحل:** بعد تنظيف شامل لكل البيانات المخالفة القديمة (راجع قسم "تنظيف البيانات" أدناه)، طُبِّقت المفاتيح
الثلاثة فعليًا على قاعدة الإنتاج بتاريخ 2026-07-19:
```sql
ALTER TABLE purchase_order_items
  ADD CONSTRAINT fk_poi_purchase_order
  FOREIGN KEY (purchaseOrderId) REFERENCES purchase_orders(id) ON DELETE RESTRICT;

ALTER TABLE warehouse_receipt_items
  ADD CONSTRAINT fk_wri_po_item
  FOREIGN KEY (purchaseOrderItemId) REFERENCES purchase_order_items(id) ON DELETE SET NULL;

ALTER TABLE inventory_transactions
  ADD CONSTRAINT fk_invtx_po_item
  FOREIGN KEY (purchaseOrderItemId) REFERENCES purchase_order_items(id) ON DELETE SET NULL;
```
**لماذا:** خط الدفاع الأخير الذي يجعل تكرار كل هذه الحوادث **مستحيلًا تقنيًا** على مستوى القاعدة نفسها، بمعزل
عن أي خطأ برمجي مستقبلي، سكربت صيانة، أو استيراد بيانات.
**⚠️ لا تُعطِّل هذه القيود ولا تُنشئ endpoint جديدًا يتجاوز `foreign_key_checks` دون خطة احتواء مماثلة.**

### 2026-07-19 — إزالة زر "تصدير PDF" العام (بلا دفعة) من صفحة تفاصيل طلب الشراء
**الملف:** `client/src/pages/purchase/PurchaseOrderDetail.tsx`
**المشكلة:** كان يوجد زرّان لتصدير PDF: زر عام أعلى الصفحة (`handleExportPdf`, بلا `batchId`) يُصدِّر كل أصناف
الطلب من كل دفعات التسعير مجتمعة، وزر آخر لكل دفعة تسعير على حدة (`handleExportBatchPdf`, بـ `batchId`)
المُسمَّى صراحة "تصدير PDF لهذه الدفعة (لطلب العهدة)". مستند العهدة المالية يجب أن يعكس دفعة واحدة مُرسَلة
فقط (تسعير جزئي أو كامل الطلب دفعة واحدة — كلاهما يُنتج `batchId` واحدًا عبر `submitPricedBatch`)، فالزر العام
كان يخالف هذا المبدأ من حيث الأساس، بصرف النظر عن حالة الطلب.
**الحل:** حُذف الزر العام بالكامل مع الدالة `handleExportPdf` والحالة `exportingPdf` غير المستخدمتين بعده. بقي
زر الدفعة (`handleExportBatchPdf`) هو المسار الوحيد لتصدير مستند العهدة، وهو محمي فعلًا (راجع البند السابق)
من عرض أصناف ملغاة أو غير مُسعَّرة أو قيد المراجعة — الأخيرة مستحيلة أصلًا ضمن أي دفعة لأن `requestItemRevision`
يرفض العملية إن كان للصنف `batchId` أو لم تكن حالته `pending`.
**لماذا:** الزر العام لا يخدم أي سيناريو عمل حقيقي (كل تسعير، جزئيًا أو كاملًا، يمر عبر آلية الدفعات نفسها)،
وإبقاؤه كان يسمح بتصدير مستند عهدة غير دقيق يجمع مبالغ من دفعات متعددة معًا.

### 2026-07-19 — إضافة "المبلغ كتابةً" (تفقيط) بجانب مبلغ العهدة في مستند PDF (نفس السطر)
**الملف:** `server/services/export/exportService.ts` — دالة `generatePurchaseRequestPDF`
**المشكلة:** سطر "المذكور عليه عهدة بمبلغ X ريال" في مستند العهدة المالية (يظهر لدور الحسابات فما بعد بعد
كتابة مبلغ العهدة) كان يعرض الرقم فقط، بلا كتابته بالحروف.
**الحل:** استُخدمت دالة **موجودة أصلًا في نفس الملف** (`amountToArabicCurrencyPhrase`، مُستخدَمة مسبقًا لتفقيط

**⚠️ خطأ وقع أثناء التنفيذ الأول وتم تصحيحه فورًا:** في المحاولة الأولى، أُضيف تطبيق تفقيط **جديد ومكرر بالكامل**
(دوال ومصفوفات بنفس الأسماء `AR_ONES`, `AR_TEENS`, `AR_TENS`, `AR_HUNDREDS`) دون التحقق أولًا من وجود تطبيق

أصلًا. **الدرس المستفاد:** قبل إضافة أي دالة مساعدة جديدة لملف موجود، يجب البحث أولًا عن وجود دالة مطابقة أو

**لماذا:** كتابة المبلغ بالحروف بجانب الرقم في نفس السطر ممارسة محاسبية معيارية في مستندات صرف العهد المالية.

### 2026-07-19 — تفعيل زر "تصدير PDF لهذه الدفعة" فور إرسال المندوب للدفعة (بدل انتظار اعتماد الحسابات)

**لماذا:** هذا المستند مرتبط بالمندوب (منشئ طلب العهدة) لا بالحسابات (الجهة المعتمِدة)، فربط تفعيله باعتماد

### 2026-07-19 — استبعاد الأصناف الملغاة والمرفوضة وغير المُسعَّرة من مستند "طلب مشتريات وصرف عهدة مالية" (PDF)
**الملف:** `server/services/export/exportService.ts` — دالة `generatePurchaseRequestPDF`
**المشكلة:**
1. الأصناف الملغاة أو المرفوضة (`rejected`, `cancelled`, `purchase_cancelled`) كانت تظهر في جدول المستند
   وتُحتسَب ضمن المبلغ الإجمالي المطلوب صرفه، حتى لو كانت تحمل تسعيرًا فعليًا من مرحلة سابقة قبل إلغائها.
2. الأصناف التي لم يُسعِّرها المندوب بعد (لا تحمل `estimatedUnitCost` ولا `actualUnitCost`) كانت تظهر أيضًا
   كسطر بتكلفة "—" بدل عدم ظهورها إطلاقًا — لا معنى لإدراج صنف في مستند "طلب عهدة مالية" قبل تحديد تكلفته.

دالة مشابهة أخرى (`generateDelegateItemsPDF`) كانت تحتوي فعلًا على فلترة صحيحة على الحالة، لكن هذه الدالة
تحديدًا لم تكن تُطبِّق أي فلتر إطلاقًا قبل هذا الإصلاح.
**الحل:**
```js
const CANCELLED_ITEM_STATUSES = new Set(["rejected", "cancelled", "purchase_cancelled"]);
const visibleItems = (allItems as any[]).filter((item: any) => {
  const isCancelled = CANCELLED_ITEM_STATUSES.has(item.status);
  const isPriced = !!(item.estimatedUnitCost || item.actualUnitCost);
  return !isCancelled && isPriced;
});
if (visibleItems.length === 0) {
  throw new Error("لا توجد أصناف مُسعَّرة وغير ملغاة لعرضها في هذا المستند");
}
// rows والإجمالي المالي (grandTotal) يُبنَيان الآن من visibleItems بدل allItems
```
فحوصات ملكية المندوب للطلب (`delegateItems`) وتحديد "من قام بالتسعير فعليًا" (`pricingDelegateId`) بقيت تعتمد
على `allItems` غير المفلترة عمدًا — الفلترة هنا خاصة بالعرض المطبوع فقط، وليست بصلاحية الوصول للمستند.
**لماذا:** عرض صنف مُلغى أو غير مُسعَّر ضمن مبلغ العهدة المطلوب صرفه يُضخِّم الرقم النهائي بلا مبرر (أو يعرض
قيمة وهمية "—") ويُربك المحاسب عند المراجعة والاعتماد.
